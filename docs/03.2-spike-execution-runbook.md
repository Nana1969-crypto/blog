# BLOG OS — Fase 03.2

## Runbook de Execução do Spike (ambiente provisionado)

> Guia operacional para **executar** a parte do Spike de Escala (03.1) que exige infraestrutura provisionada — os eixos que o harness local **não** pode medir: **H3** (cache/CDN), **H4** (Core Web Vitals de campo), **H7** (rate limit/custo de API) e **H8** (workflow do CMS), além do **tempo de build incremental real**.
> Status: **Runbook pronto para execução** · Versão 1.0 · Depende de: `03`, `03.1`, harness em [`../spike/`](../spike/README.md), ADRs `0001–0004`
> Escopo: **procedimento e template de relatório**. Não é código de produção. O que for construído aqui (app de render, importador) é **descartável**.

---

## 0. Antes de começar — regras inegociáveis

1. **Não fabrique números.** Toda métrica deste runbook vem de medição real no ambiente. Célula sem medição fica `PENDENTE`, nunca estimada.
2. **É throwaway.** App de render, importador e scripts de carga são descartáveis (princípio §0 do spike). Nada vira produção.
3. **Meça p95/p75, não média** (03.1 §11.5). A cauda define a experiência e o SEO.
4. **Meça a curva, não o ponto** (03.1 §11.2): 1k → 10k → 100k → 1M. O sinal está na inclinação.
5. **Um eixo estrutural reprovado reprova o conjunto** (03.1 §9). Não "arredonde" para aprovar.

## 4. Execução por eixo

### 4.1 Build incremental real (H1 — confirmação, o teste mais importante)

**Procedimento:**
1. Build inicial (cold) do catálogo de 100k; registrar tempo total e memória de pico.
2. Publicar/editar **1 artigo**; medir só o trabalho incremental (regeneração + purga).
3. Publicar em lote (100 de uma vez) — pico editorial.
4. Repetir 1–2 em 1k/10k/100k/1M e **plotar a curva**.

**Alvos (03.1 §5.2):**
| Métrica | Alvo | Registro |
|---|---|---|
| Build incremental (1 artigo) | < 30s e **~constante** vs. catálogo | `PENDENTE` |
| Páginas de índice regeneradas/publicação | só as afetadas (comparar com fan-out do harness: p95≈25) | `PENDENTE` |
| Build inicial 100k | tolerável e paralelizável | `PENDENTE` |
| Memória de pico | dentro do runner | `PENDENTE` |
| ISR miss (regeneração sob demanda) | < 1s/página | `PENDENTE` |

> **Cruzamento com o harness:** o fan-out medido localmente (p95≈25 páginas) prevê o trabalho de regeneração por publicação. Se o build real regenerar **muito mais** que isso, a implementação de invalidação diverge do modelo — investigar antes de aprovar.

### 4.2 CDN e Cache (H3)

**Procedimento (03.1 §6):**
1. Carga de leitura realista; medir **cache hit ratio** (HTML) e TTFB de múltiplas regiões.
2. Publicar 1 artigo; confirmar **invalidação seletiva** (só rotas do grafo) e tempo de propagação global.
3. Teste de **stampede**: muitos usuários no primeiro hit pós-regeneração.
4. **Caso difícil (ADR-0003):** mudar o título de um artigo linkado por ~500 páginas; observar o fan-out de âncora e decidir política (runtime vs. build).

**Alvos:**
| Métrica | Alvo | Registro |
|---|---|---|
| Cache hit ratio (HTML) | > 95% | `PENDENTE` |
| Invalidação por publicação | só rotas afetadas (sem purga global) | `PENDENTE` |
| Propagação global | < 60s | `PENDENTE` |
| TTFB global (p75) | < 200ms | `PENDENTE` |
| Stampede | coalescência de origin | `PENDENTE` |

### 4.3 Performance / CWV de campo (H4)

**Procedimento (03.1 §7):** medir em **artigo real de pior caso** (guia longo com tabelas, imagens, code, embeds), em dispositivo modesto + rede 3G/4G. Lab (Lighthouse CI) + campo (RUM).

**Alvos (p75 campo):**
| Métrica | Alvo | Registro |
|---|---|---|
| LCP | < 2.0s | `PENDENTE` |
| CLS | < 0.1 | `PENDENTE` |
| INP | < 200ms | `PENDENTE` |
| TTFB | < 200ms | `PENDENTE` |
| JS inicial (rota de leitura) | < ~100KB gz | `PENDENTE` |
| Lighthouse Perf | ≥ 95 | `PENDENTE` |

> Falha em **mobile modesto/3G** é reprovação, mesmo com desktop verde (Princípio 31).

### 4.4 API do CMS (H7)

**Procedimento (03.1 §8):** medir chamadas necessárias para build/regeneração de 100k; bater no rate limit; projetar custo mensal (build + ISR + previews + operação) e a curva 100k→1M.

**Alvos:**
| Métrica | Alvo | Registro |
|---|---|---|
| Latência de leitura (p95) | < 100ms | `PENDENTE` |
| Build inicial sem 429 | sim (bulk/backoff) | `PENDENTE` |

| Bulk/export disponível | sim | `PENDENTE` |
| Webhook de publicação confiável | sim (+ fallback por tempo) | `PENDENTE` |

### 4.5 Workflow do CMS (H8)

**Procedimento (03.1 §4 + Fase 04):** exercitar criação/edição/publicação/agendamento, permissões com milhares de autores, busca do editor em 100k, preview de rascunho estruturado.

**Alvos:**
